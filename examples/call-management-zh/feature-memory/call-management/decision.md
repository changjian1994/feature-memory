# Decisions

## 2026-06-09：第一期保持只读

- 决策：第一期只提供观测和排障能力。
- 原因：控制动作需要更严格的权限、审计、幂等和回滚设计。
- 取舍：用户暂时不能在该模块挂断或重试通话，但 feature 能以更低风险更快落地。

## 2026-06-09：列表以会话表为主

- 决策：通话列表查询基于通话会话表。
- 原因：会话表是稳定的生命周期数据源，避免从事件表做高成本聚合。
- 取舍：事件细节仍然保留，但只在详情时间线中展示。

## 2026-06-09：洞察数据按需加载

- 决策：转写、质检和录音通过按需 `insights` 接口聚合。
- 原因：这些数据体积较大，不适合每一行列表默认加载。
- 取舍：列表首屏更快，详情页在需要时额外请求。

## 2026-06-09：录音使用带鉴权的 blob 获取

- 决策：前端通过带鉴权的 HTTP client 获取录音 blob，再创建 object URL 播放。
- 原因：浏览器原生媒体请求可能不会携带业务鉴权头。
- 取舍：前端需要管理 object URL 生命周期，但不会暴露对象存储直连地址。

## 2026-06-09：列表和详情使用独立 WebSocket 生命周期

- 决策：列表刷新和详情刷新使用独立 WebSocket。
- 原因：用户只看列表时，不应该持续加载事件、转写、质检和录音元数据。
- 取舍：前端多维护一个连接生命周期，但后端负载更可控。
