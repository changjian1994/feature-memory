# 通话管理设计

## 背景

通话平台的会话、事件、转写、质检和录音元数据分布在多个系统中。管理后台需要帮助用户查看通话和排查问题，但不应该在第一期变成控制面。

## 推荐方案

- 使用通话会话表作为列表和详情基础数据源。
- 使用通话事件表展示事件时间线和解析录音元数据。
- 使用按需 `insights` 接口聚合转写、质检和录音摘要。
- 录音文件通过后端代理读取，由后端校验归属和鉴权。
- WebSocket 只刷新列表和详情的轻量状态，不传输音频内容或大 JSON。

## 数据加载规则

- 列表页：只加载轻量字段。
- 详情抽屉：基础信息和被用户打开的大字段按需加载。
- 事件时间线：默认展示摘要，原始 payload 放在“查看 JSON”入口后。
- 录音：只展示可播放文件，录音生命周期事件作为状态摘要。
- 转写和质检：仅在用户打开详情或展开行时加载。

## 风险

- 列表查询加载大 payload 会影响性能。
- 直接暴露对象存储 URL 可能泄漏实现细节或凭证边界。
- 浏览器原生媒体请求可能绕过前端鉴权拦截器。
- debug 日志和 feature memory 不能记录密钥或生产数据原文。
