# 隐私与安全边界

`feature-memory` 会在目标项目中保存 Markdown 形式的开发上下文。这些文件可能包含架构说明、接口描述、调试结论和交接信息。写入、分享或提交这些文件前，必须按项目文档的安全标准检查。

## 禁止记录

不得将以下内容写入 `feature-memory/` 文档：

- 密钥、私钥、访问凭证、API Key、密码或账号凭据。
- Token、Cookie、Session、JWT、Refresh Token 或 Authorization Header。
- 个人数据、用户标识、客户数据或生产数据导出。
- 内部域名、私有网络地址、服务发现名称或非公开基础设施细节。
- 未脱敏的生产日志、崩溃转储、请求体、数据库行或链路追踪数据。
- 未获准公开的业务规则、路线图、产品计划或内部策略。

## 推荐做法

- 优先记录结论，不复制原始敏感数据。
- 记录日志、接口样例或数据库片段前先脱敏。
- 使用 `<redacted-token>`、`<internal-domain>`、`<user-id>`、`<sample-id>` 等占位符。
- 只有在确认安全时，才把 debug 输出保存到 `feature-memory/<feature-name>/debug/.../output/`。
- 提交或分享记忆文档前，先检查是否包含敏感内容。

## Git 提交策略

目标项目中的 `feature-memory/` 是否提交到 Git 由项目决定：

- 公开或开源仓库：除非所有内容都已审查和脱敏，否则不要提交 `feature-memory/`。
- 内部仓库：遵循团队的文档和数据处理规范。
- 敏感排障会话：可将 `feature-memory/` 加入目标项目的 `.gitignore`，只分享脱敏后的摘要。

## Agent 行为

使用此 skill 的 Agent 应：

- 在保存疑似敏感日志、接口样例或数据库细节前先确认。
- 用摘要替代原文复制。
- 将不确定信息标记为“假设”或“待确认问题”。
- 对一次性小修改或低风险任务，除非用户明确要求，否则不主动创建记忆文件。
